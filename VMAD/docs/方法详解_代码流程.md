# VMAD 流程详解

> 本文档简明描述 VMAD 的完整工作流程，对每个关键概念附代码片段，并提供学术化解释和大白话解释。

---

## 总览：两阶段架构

VMAD 的核心任务是**视频 Prompt 反演**：给定一段参考视频和一个冻结的文生视频模型，找到最优的输入条件，让模型忠实地复现这段视频。

整体分为两个阶段：

```
Extract（提取）: 参考视频 --> [Phase A + Phase B + Phase C] --> Motion Asset
Apply（应用）:   Motion Asset + 新 prompt --> [Layer 1 + Layer 2 + Layer 3] --> 生成视频
```

Extract 阶段产出一个 Motion Asset（运动资产），包含三个文件：

- `eta_motion`：反演噪声（来自 Flow Matching Inversion）
- `delta_e`：embedding 残差（来自 Velocity Field Matching 优化）
- `motion_text`：文本描述（来自 Token Decode，已证明无效）

目前实验验证，只有 **Phase B 的 delta_e** 和 **Layer 2 的 Hook 注入** 是真正有效的。最佳配置为 `--alpha 0.005 --no-blend --no-token_decode`。

### Phase A: Flow Matching Inversion（视频 → 反演噪声）

#### Step A1: VAE 编码（视频像素 → 潜空间表示）

```python
# pipeline.py L712-713
ref_norm = normalize_video(ref_video).unsqueeze(0)  # [0,1] -> [-1,1]
ref_latents = encode_video_to_latents(pipe, ref_norm, device)
# ref_latents shape: (1, 16, 21, 60, 104) [B, C_latent, F_latent, H_latent, W_latent]
```

VAE 编码的具体实现（`flow_matching.py` L29-62）：

```python
def encode_video_to_latents(pipe, video_tensor, device):
    vae = pipe.vae
    video_tensor = video_tensor.to(device=device, dtype=vae.dtype)

    with torch.no_grad():
        latent_dist = vae.encode(video_tensor)
        if hasattr(latent_dist, "latent_dist"):
            latents = latent_dist.latent_dist.sample()
        elif hasattr(latent_dist, "sample"):
            latents = latent_dist.sample()
        else:
            latents = latent_dist

    # 应用 scaling factor
    if hasattr(vae.config, "scaling_factor"):
        latents = latents * vae.config.scaling_factor

    return latents
```

**学术解释**：VAE（Variational Autoencoder，变分自编码器）是一种将高维数据压缩到低维潜空间（latent space）的生成模型。Wan2.1 使用的 3D VAE 将视频从像素空间 (B, 3, 81, 480, 832) 编码为潜空间表示 (B, 16, 21, 60, 104)，空间维度压缩 8 倍（480/8=60, 832/8=104），时间维度压缩约 4 倍（81/4≈21），通道数从 RGB 3 通道扩展到 16 个潜通道。编码后乘以 `scaling_factor` 以规范化潜变量的分布范围。

**大白话**：VAE 就是一个"压缩器"。一段 81 帧、480x832 的视频数据量巨大（81x3x480x832 ≈ 1 亿个数字），直接处理太贵了。VAE 把它压缩成一个小得多的"浓缩版"（16x21x60x104 ≈ 200 万个数字），信息量少了 50 倍，但保留了视频的核心结构。后续所有操作都在这个"浓缩版"（latent）上进行，最后生成视频时再用 VAE 的解码器把它还原回像素。

#### Step A2: Flow Matching Inversion（潜空间 → 反演噪声）

```python
# pipeline.py L716-728

prompt_embeds = self._encode_prompt("")

    guidance_scale=1.0,

)
eta_inv = inverter.invert(ref_latents, prompt_embeds)
```

Euler 正向积分的具体实现（`flow_matching.py` L142-178）：

```python
def invert(self, latents, prompt_embeds):
    """Euler 一阶反演: z0 (t=0) -> eta (t=1)"""
    dt = 1.0 / self.num_steps  # 每步 0.02
    x = latents.clone()

    for i in range(self.num_steps):  # 50 步
        t_norm = i * dt                    # t 从 0 递增到 0.98
        t_model = torch.tensor(t_norm * 1000.0, device=self.device)  # 转为 Wan2.1 的 [0,1000] 格式

        # 预测速度场 v_theta(x_t, t, c="")
        v = self._model_forward(x, t_model, prompt_embeds)

        # Euler 步进: x_{t+dt} = x_t + v * dt
        x = x + v * dt

    return x  # 这就是 eta_inv
```

**学术解释**：Flow Matching 是一种将数据分布和噪声分布之间的映射建模为 ODE（常微分方程）的生成范式。在 Rectified Flow 框架下，数据 z₀ 和噪声 η 之间通过线性插值路径连接：x_t = (1-t)·η + t·z₀，对应的速度场为 v*(t) = z₀ - η（常数，不依赖 t）。Flow Matching Inversion 是生成过程的逆操作：从 z₀ 出发，沿模型预测的速度场 v_θ(x_t, t, c) 正向积分 ODE（t: 0→1），得到"如果从该噪声出发正向生成，理论上会还原 z₀"的反演噪声 η_inv。使用空字符串作为条件（无条件反演）是为了让 η_inv 编码参考视频的全部信息，不与特定文本条件耦合。

**大白话**：想象文生视频模型是一个"画师"，他的创作过程是从一团随机涂鸦（噪声）开始，一步步"擦除"杂乱变成清晰的画面。Flow Matching Inversion 做的是**倒放**这个过程——拿一段已有的视频，倒推回去找到"如果画师要画出这段视频，他最开始拿到的涂鸦应该长什么样"。我们用 50 小步模拟这个倒推过程（Euler 方法），每一步问模型"你在这个时间点会往哪个方向画"，然后沿相反方向走一小步。50 步之后就得到了 η_inv——一个"定制化的起始噪声"。

---

#### 为什么用空字符串做反演条件？

代码中用 `self._encode_prompt("")` 而非参考视频的 caption：

**学术解释**：无条件反演使 η_inv 独立于任何文本语义，编码的是"在无语义指导下从当前点到达 z₀ 所需的全部信息"。这为后续 Velocity Matching 留出最大的优化空间——delta_e 需要承载的信息量最大化。若使用有条件反演，η_inv 会与 caption 耦合，delta_e 优化的余量变小。

**大白话**：如果我们在倒推时告诉模型"这是一只猫在跳"，那倒推出的噪声里已经"偷偷藏了"这些文字信息。后面优化 delta_e 时就没什么可学的了。用空白条件倒推，噪声里只有纯粹的视觉信息（猫的样子、跳的轨迹等），这样 delta_e 才能学到最丰富的内容。

### Phase B: Velocity Field Matching（核心有效步骤）

这是 VMAD 最关键的步骤。目标：优化一个 delta_e，注入到模型的文本条件中，让模型"记住"参考视频的运动。

#### Step B1: 准备输入

```python
# pipeline.py L410-432
# VAE 编码参考视频 -> z0
ref_norm = normalize_video(ref_video).unsqueeze(0)
z0 = encode_video_to_latents(self.pipe, ref_norm, self.device)

# T5 编码 caption -> e0（初始 text embedding）
e0 = self._encode_prompt(caption)
# e0 shape: (1, 512, 4096) [B, SeqLen, HiddenDim]

# 如果有 eta_inv 就用它构建目标速度，否则用随机噪声
eta_for_opt = eta_inv if eta_inv is not None else torch.randn_like(z0)
```

**关于 T5 编码**（`pipeline.py` L906-932）：

```python
def _encode_prompt(self, prompt: str) -> torch.Tensor:
    """Encode text to embedding via T5 text encoder.
    Wan2.1: output shape is (1, 512, 4096) -- UMT5-XXL hidden_size=4096."""
    result = self.pipe.encode_prompt(
        prompt=prompt, device=self.device,
        num_videos_per_prompt=1,
        do_classifier_free_guidance=False,
        max_sequence_length=512,
    )
    return result[0] if isinstance(result, tuple) else result
```

**学术解释**：T5（Text-to-Text Transfer Transformer）是一个预训练语言模型，Wan2.1 使用 UMT5-XXL 作为文本编码器。它将自然语言 prompt 编码为形状 (1, 512, 4096) 的连续向量序列——512 个位置，每个位置一个 4096 维的向量。这个向量序列通过 cross-attention 注入 DiT（Diffusion Transformer），指导视频生成的方向。

**大白话**：T5 就是一个"翻译器"，把人类的文字翻译成模型能理解的"数字语言"。一段 prompt 被翻译成 512 个"词向量"，每个词向量有 4096 个维度。生成视频时，模型会不断"查看"这些词向量来决定画什么。

#### Step B2: 优化循环（核心！）

```python
# pipeline.py L435-454
matcher = PositionAwareVelocityMatcher(
    pipe=self.pipe,
    T_m=1.0,            # 全时间步匹配（复现模式）
    num_opt_steps=200,  # 优化步数
    lr=1e-3,            # Adam learning rate
    lambda_dis=0.0,     # 不做内容解耦（复现模式）
    lambda_pos=0.01,    # Position-aware 正则化权重
    position_aware=True,
)

opt_result = matcher.optimize(z0=z0, e0=e0, eta_inv=eta_for_opt)
delta_e = opt_result["delta_e"]  # shape: (1, 512, 4096)
```

优化循环内部（`velocity_matching.py` L367-456）：

```python
delta_e = torch.zeros_like(e0, requires_grad=True)  # 初始化为全零
optimizer = torch.optim.Adam([delta_e], lr=1e-3)

for step in range(num_opt_steps):
    optimizer.zero_grad()

    # 1. 随机采样一个时间步 t ∈ [0, T_m=1.0]
    t_norm = torch.rand(1, device=self.device) * self.T_m
    t_model = t_norm * 1000.0  # 转为 Wan2.1 时间步格式

    # 2. 构造 ODE 轨迹上的中间状态
    x_t = (1 - t_norm) * eta_inv + t_norm * z0

    # 3. 计算目标速度（封闭解！）
    v_star = z0 - eta_inv

    # 4. 模型在当前条件下预测的速度
    e_current = e0 + delta_e
    v_pred = self._model_forward(x_t, t_model, e_current)

    # 5. MSE 损失：让预测速度逼近目标速度
    loss_vel = ((v_pred - v_star) ** 2).mean()

    # 6. Position-aware 正则化
    loss_pos = self._compute_position_regularization(delta_e, position_weights)

    # 7. 总损失 + 反向传播
    loss_total = loss_vel + 0.01 * loss_pos
    loss_total.backward()

    # 8. Position-aware 梯度缩放
    if self.position_aware and delta_e.grad is not None:
        grad_scale = 1.0 / (position_weights + 0.1)
        grad_scale = grad_scale / grad_scale.mean()
        delta_e.grad.data *= grad_scale.unsqueeze(0).unsqueeze(-1)

    # 9. Adam 更新 + Cosine 学习率衰减
    optimizer.step()
    scheduler.step()
```

下面逐步解释：

---

##### Step 2 解释：x_t = (1-t)·η_inv + t·z₀

**学术解释**：在 Rectified Flow 框架中，数据（z₀）和噪声（η_inv）之间通过线性插值路径连接。x_t 是 ODE 轨迹上时间 t 处的中间状态——t=0 时 x_t = η_inv（纯噪声），t=1 时 x_t = z₀（纯数据），中间是两者的线性组合。这条直线路径是最优传输路径（optimal transport），使得模型的速度场在理论上为常数。

**大白话**：想象从一团噪声到最终视频之间有一条直线。t 就是"走了多远"——t=0 时你还在起点（噪声），t=1 时你到终点（视频），t=0.5 时你走了一半，是噪声和视频各占一半的"半成品"。我们每次随机挑一个"走到哪了"的位置来练习。

---

##### Step 3 解释：v* = z₀ - η_inv

**学术解释**：这是 Rectified Flow 的核心性质——最优传输路径上的速度场是常数 v*(t) = data - noise = z₀ - η_inv，不依赖时间 t。这意味着在轨迹上的任何一点，"正确的前进方向"都是同一个固定向量。这是一个**封闭解**（closed-form solution），不需要采样估计或近似计算。

**大白话**：因为路径是直线，所以不管你走到哪了，"正确方向"永远是同一个——从起点指向终点的方向（终点 - 起点 = z₀ - η_inv）。这是一个可以直接算出来的答案，不需要猜。这比其他方法（比如 SDS 没有明确目标）强太多了。

---

##### Step 4 解释：v_pred = model(x_t, t, e0 + delta_e)

**学术解释**：冻结模型参数 θ，只通过改变条件输入 c = e₀ + Δe 来改变模型的预测方向。这是 Conditioning Inversion 的核心——在模型的"条件超平面"上进行优化搜索，而非修改模型本身。

**大白话**：模型（画师）的技术是固定的，我们不改他的画功。我们改的是给他的"指令"（embedding）。通过微调指令里的一小段（delta_e），让他在每个时间点画的方向都尽量指向我们想要的目标。

---

##### Step 5 解释：loss = ||v_pred - v*||²

**学术解释**：标准的 Velocity Field Matching 目标，等价于 Flow Matching 的训练损失。区别在于：训练时优化模型参数 θ 使其预测 v* for all inputs，而 VMAD 固定 θ，优化 Δe 使其在特定参考视频上预测 v*。

**大白话**：比较模型实际画的方向 (v_pred) 和理想方向 (v*)，算它们差多远（距离的平方）。这个差距就是"损失"，优化的目标是让它尽可能小——让模型的每一笔都指向目标视频。

---

##### Step 8 解释：Position-Aware 梯度缩放

Position weights 的初始化（`velocity_matching.py` L184-223）：

```python
def _initialize_position_weights(self, seq_len):
    weights = torch.ones(seq_len, device=self.device)
    weights[0] = 0.1   # Position 0: 低正则化 = 高自由度
    if seq_len > 1:
        weights[-1] = 0.3  # 尾部: 中等
    # 中间位置: 抛物线 U-shape
    for i in range(1, seq_len - 1):
        dist_from_edge = min(i, seq_len - 1 - i)
        weights[i] = 0.5 + 0.5 * (dist_from_edge / center)
    return weights
```

梯度缩放：`grad_scale = 1.0 / (position_weights + 0.1)` — 权重越小的位置（如 position 0），梯度放大越多。

**学术解释**：DiT 架构的 cross-attention 存在 U-shape position bias（类似 LLM 中的 Attention Sink 现象）——position 0 接收的注意力权重是中间位置的 10-15 倍。这意味着 Δe 在 position 0 的变化对生成结果影响最大。Position-Aware 策略据此将优化力度集中在高影响力位置：对 position 0 给予最低的正则化惩罚（允许自由变化）、最高的梯度放大（加速收敛），而对中间位置施加更强约束。

**大白话**：模型在读取文本条件时，对"第一个位置"特别敏感（好比你读一篇文章，对标题的印象最深）。所以我们把优化精力集中在这个位置——允许它大幅变化、放大它的学习信号。中间位置影响小，就别浪费精力去改了。这样收敛更快。

### Phase C: Token Decode（当前关闭）

```python
# pipeline.py L481-493
if cfg.use_token_decode and delta_e is not None:
    token_decode_result = self.token_decoder.decode(
        delta_e=delta_e, e0=e0, z0=z0, base_caption=caption
    )
    motion_text = token_decode_result["motion_text"]
    # 结果通常是乱码，如 "ayarak guerre politics 熙 カラー"
```

**学术解释**：Token Decode 尝试将连续的 Δe 投影到 T5 词表的离散 token 序列，使结果可解释、可跨模型迁移。其失败根因在于 Δe 处于 T5 的**输出 hidden states 空间**（经过 24 层 transformer 变换后的空间），而最近邻搜索在 T5 的**输入 embedding 空间**进行——两个空间维度相同但拓扑结构完全不同。

**大白话**：试图把优化出的"数字密码"翻译回人类文字。但翻译器用错了字典（在输入空间找最近邻，但 delta_e 在输出空间），所以翻出来全是乱码。

**当前状态**：❌ 有害（CLIP -0.016），用 `--no-token_decode` 关闭。

### Layer 2: delta_e Embedding Hook（唯一有效的注入层）

```python
# pipeline.py L643-648
```

Hook 机制的具体实现（`pipeline.py` L828-904）：

```python
def _generate_with_embedding_hook(self, prompt, latents, generator, delta_e, strength):
    text_encoder = self.pipe.text_encoder

    def text_encoder_hook(module, input, output):
        """在 T5 encoder 输出时拦截，加上 delta_e"""
        if isinstance(output, tuple):
            hidden_states = output[0]
        else:
            hidden_states = output

        # 核心注入：e_final = e_original + strength * delta_e
        de = delta_e.to(device=hidden_states.device, dtype=hidden_states.dtype)
        hidden_states = hidden_states + strength * de

        if isinstance(output, tuple):
            return (hidden_states,) + output[1:]
        return hidden_states

    # 注册 hook -> 正常跑 pipeline -> 移除 hook
    hook_handle = text_encoder.register_forward_hook(text_encoder_hook)
    try:
        video = self._generate(prompt, latents, generator)
    finally:
        hook_handle.remove()

    return video
```

**学术解释**：Hook 机制在 T5 encoder 的 forward pass 完成后拦截输出，将 Δe 以极小强度（α=0.005）叠加到 encoder hidden states 上：`e_final = e_original + 0.005 * Δe`。选择 Hook 而非直接传 `prompt_embeds` 的原因是保持 Classifier-Free Guidance (CFG) 的正常工作——CFG 需要同时编码正例和负例 prompt（`v_final = v_uncond + scale * (v_cond - v_uncond)`），如果直接传 `prompt_embeds`，diffusers 会跳过负例编码。Hook 方式让 pipeline 完全走正常路径，只在正例 prompt 编码时"偷偷"加一个微扰动。

**大白话**：生成视频时，模型会先把 prompt 翻译成数字（T5 编码），然后用这些数字来指导画画。我们在翻译完成的那一刻"偷偷"往数字里加了一点点修正（只有原始值的千分之五），让模型的画笔方向稍微偏向参考视频的运动模式。因为加得很少，所以不会破坏画面质量，但足以让生成的视频更像参考视频的运动。

---

#### 为什么 alpha 要这么小（0.005）？

**学术解释**：Velocity Matching 优化后 ||Δe|| ≈ 36，而 ||e₀|| ≈ 180。若直接注入（alpha=1），Δe 占 e₀ 范数的 20%，会过度偏移条件空间导致生成崩溃。alpha=0.005 使实际注入 ||0.005*Δe|| ≈ 0.18，仅占 e₀ 的千分之一，属于微扰动范围。

**大白话**：优化出的 delta_e "能量"太强了，直接加进去模型会"蒙圈"——画面崩溃出色块。所以要极大幅度缩小它（除以 200），让它变成一个很轻很轻的"暗示"，而不是一个"强制命令"。

---

### Layer 3: Noise Prior Blending（当前关闭）

```python
# pipeline.py L659-667
if cfg.use_blend:
    latents = self._asset_manager.apply_noise_prior(
        asset, alpha=cfg.alpha, strength=strength, generator=generator
    )
    # 混合公式: latents = sqrt(alpha*s) * eta_motion + sqrt(1-alpha*s) * eta_random
else:
    latents = None  # 让 pipeline 内部自己生成随机噪声
```

**学术解释**：设计意图是用反演噪声 η_inv（包含参考视频结构信息）部分替代随机噪声作为生成起点，利用 `sqrt(α)·η_inv + sqrt(1-α)·η_random` 的方差保持公式（保证混合后仍为标准正态分布）来引导 ODE 轨迹经过目标 latent 附近。

**大白话**：想用"倒推出的起点"替代随机涂鸦作为画师的起始画布，这样画师一开始就在"正确的位置"，画出来自然更像目标。

**当前状态**：❌ 严重有害（CLIP -0.10~-0.15）。根因是 `torch.randn` 和 pipeline 内部的 `randn_tensor` 即使使用同一个 generator 也产生不同随机数（内存布局不同），导致传入外部噪声后生成的视频和"不开 blend"时完全不是同一个东西。这是实现 bug 而非理论问题。

**大白话**：模型从一团随机噪声开始，分 30 步逐渐"画清楚"画面。每一步都参考文本条件来决定方向，同时用一个"无条件"的对照来增强文字的指导效果（这就是 CFG）。因为流程完全确定，同一个随机种子一定能画出一模一样的视频。

## 三、当前实际生效的精简流程

去掉所有无效部分后，VMAD 的实际有效流程是：

```
输入：参考视频 + V4 caption

Extract 阶段：
  1. VAE 编码参考视频 -> z0
  2. Flow Matching Inversion(z0, 空条件, 50步 Euler) -> eta_inv
  3. T5 编码 V4 caption -> e0
  4. Velocity Field Matching(z0, e0, eta_inv, 200步 Adam) -> delta_e

Apply 阶段：
  1. T5 编码新 prompt -> e0_new（pipeline 内部完成）
  2. Hook: e0_new += 0.005 * delta_e
  3. Pipeline 正常生成视频（latents=None，30步去噪）

输出：生成视频
```

对应命令：
```bash
python run_batch_extract.py \
    --caption-dir <V4_CAPTIONS> \
    --alpha 0.005 \
    --no-blend --no-token_decode --no-svd --no-disentangle \
    --content SELF
```

最优结果（10 样本验证）：CLIP=0.9446, XCLIP=0.7541。

## 四、各步骤有效性总结

| 步骤 | 阶段 | 有效性 | 说明 |
|------|------|--------|------|
| VAE 编码 | Extract 前置 | ✅ 必要 | 将视频压缩到可计算的 latent 空间 |
| Flow Matching Inversion | Extract Phase A | ⚠️ 间接有用 | 提供 eta_inv 给 VFM 的 v* 计算 |
| SVD 分解 | Extract Phase A | ❌ 关闭 | 当前 `--no-svd` 跳过 |
| **Velocity Field Matching** | **Extract Phase B** | **✅ 核心有效** | **唯一真正有用的优化步骤** |
| Position-Aware 梯度缩放 | Extract Phase B | ❓ 未独立验证 | 理论合理但无对照实验 |
| Token Decode | Extract Phase C | ❌ 有害 | 产出乱码，CLIP -0.016 |
| Text Fusion | Apply Layer 1 | ❌ 有害 | 拼接乱码污染 prompt |
| **delta_e Hook** | **Apply Layer 2** | **✅ 核心有效** | **alpha=0.005 时 CLIP+0.025, XCLIP+0.022** |
| Noise Prior Blend | Apply Layer 3 | ❌ 严重有害 | 实现 bug，CLIP -0.10~-0.15 |
