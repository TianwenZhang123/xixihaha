## 二、Bug 修复与 Pipeline 等价性验证

**时间**：2025-05-31

### 发现的 Bug

| # | Bug | 位置 | 影响 | 修复 |
|---|-----|------|------|------|
| 1 | `apply_noise_prior` 无条件调用 | pipeline.py L657 | generator 状态被额外 `torch.randn` 消耗，后续 diffusion 随机序列偏移 | 加 `if cfg.use_blend:` |
| 2 | `motion_text` 无条件拼接 | pipeline.py L630 | 即使 `--no-token_decode`，仍拼接垃圾 token 到 prompt | 加 `and cfg.use_token_decode` |

**Token Decode 产生乱码的根因**：Token Decoder 在 T5 **输入 embedding 空间**做最近邻搜索，但 Δe 存在于 T5 **输出 hidden states 空间**（经过 12 层 Transformer）。两个空间几何结构完全不同。代码有 `_build_output_codebook()` 可在正确空间搜索但未被调用。

### 等价性验证（sample #7）

| 实验配置 | orig_gen_clip | orig_gen_xclip |
|----------|--------------|----------------|
| P-Flow baseline | 0.9159 | 0.7108 |
| P-Flow reseed（确认可复现） | 0.9159 | 0.7108 |
| VMAD 修复前（Bug 1+2） | 0.8007 | — |
| VMAD 修复 Bug1，Bug2 残留 | 0.8986 | 0.6384 |
| **VMAD 三层全关（两 bug 修复）** | **0.9150** | **0.7077** |

**结论**：修复后 VMAD pipeline 在三层全关时与 P-Flow 等价（CLIP 差异 < 0.001）。

### 3.1 Baseline Caption + Strength 扫描（sample #7）

| Alpha | orig_gen_clip | orig_gen_xclip | Δ CLIP vs 基线 |
|-------|--------------|----------------|----------------|

| 0.05 | 0.8999 | 0.5925 | -0.0151 |
| 0.1 | 0.9016 | 0.5670 | -0.0134 |

**分析**：倒 U 形曲线，α=0.01 时 CLIP 超过 P-Flow baseline。Δe 确实编码了有用运动信息。

### 3.2 V4 Caption × Layer 2 消融（sample #7）

2×2 矩阵验证 caption 质量与 Δe 注入的叠加关系：

| 编号 | Caption | 层配置 | orig_gen_clip | orig_gen_xclip |
|------|---------|--------|--------------|----------------|
| A | Baseline | 三层全关 | 0.9150 | 0.7077 |
| B | Baseline | L2 α=0.01 | 0.9192 | 0.6987 |
| **C** | **V4** | **三层全关** | **0.9397** | **0.7260** |
| **D** | **V4** | **L2 α=0.01** | **0.9394** | **0.7486** |

**核心结论**：

1. V4 caption 主攻 CLIP（+0.0247），Δe 主攻 XCLIP（在 V4 基础上 +0.0226）
2. 两者互补叠加但贡献维度不同——CLIP 被 V4 caption 饱和后，Δe 的残差修正被"释放"到时序维度
3. D 配置的 XCLIP=0.7486 超过 P-Flow 所有实验

## 四、Layer 3 (Noise Prior) 消融 — 结论：有害

**时间**：2025-06-01

| 编号 | Caption | 层配置 | orig_gen_clip | orig_gen_xclip |
|------|---------|--------|--------------|----------------|
| A | Baseline | 三层全关 | 0.9150 | 0.7077 |
| E | Baseline | L3 only (α=0.001) | 0.8097 | 0.6407 |
| F | Baseline | L2+L3 (α=0.01) | 0.7952 | 0.7025 |
| C | V4 | 三层全关 | 0.9397 | 0.7260 |
| G | V4 | L3 only (α=0.001) | 0.7993 | 0.6782 |
| H | V4 | L2+L3 (α=0.01) | 0.7875 | 0.6952 |

**结论：Layer 3 在当前实现中有害（CLIP 暴跌 -0.10~-0.15）。**

**根因分析**：`apply_noise_prior()` 调用 `torch.randn(generator=generator)` 消耗了共享 generator 的状态。但更深层问题是 **latents 生成路径差异** —— 外部用 `torch.randn` 生成的噪声与 pipeline 内部 `prepare_latents` 用 `randn_tensor` 生成的噪声，即使相同 generator 状态也会产生不同结果（因为 `randn_tensor` 的实现可能涉及分块生成或不同的内存布局）。

**重要修正**：此前"generator state offset"理论**部分不正确**。`FlowMatchEulerDiscreteScheduler` 是确定性调度器（`sigma=0`），在去噪循环中**不消耗 generator**。真正的问题是：
1. pipeline 的 `prepare_latents` 内部用 `randn_tensor` 生成初始噪声
2. 外部 `apply_noise_prior` 用 `torch.randn` 生成混合后的噪声
3. 即使 generator 状态相同，这两条路径产生的随机数**完全不同**（不只是偏移，是完全不同的数）

**修复方案（未实施）**：不再外部生成噪声，而是让 pipeline 正常生成初始噪声（通过 `prepare_latents`），然后在外部将 eta_motion blend 进去：
```python
# 让 pipeline 正常生成 latents（保证路径一致）
latents = pipe.prepare_latents(...)  # 内部用 randn_tensor
# 外部 blend eta_motion
latents = sqrt(alpha) * eta_motion + sqrt(1-alpha) * latents
```

**决策：当前阶段放弃 Layer 3，专注 Layer 2 + V4 caption。Layer 3 需要工程重构才能正确工作。**

## 六、技术架构备忘

### VMAD Apply 流程（修复后）

```
content_prompt
     │
     ▼
[Layer 1] if use_token_decode: prompt += motion_text  ← 当前产生乱码，建议关闭
     │
     ▼
[Layer 2] if use_velocity: hook(encoder_output += effective_strength × Δe)
     │                     effective_strength = strength × alpha
     ▼
[Layer 3] if use_blend: latents blend  ⚠️ 有害！路径差异导致结果偏移！不应使用！
     │
     ▼
[Generate] pipe(prompt=full_prompt, latents=latents, generator=generator)
```

### cfg.alpha 的双重耦合

同一个 `cfg.alpha` 参数同时控制：
- Layer 2：`effective_strength = strength × alpha`（Δe 注入强度）
- Layer 3：`alpha_scaled = alpha × strength`（noise blending 比例）

这是设计问题——两层应该有独立参数。当前因 Layer 3 关闭所以不影响实验，但后续若修复 Layer 3 需要解耦。

### P-Flow vs VMAD Seed 策略差异

- P-Flow：`seed = cfg.seed + sample_id`（每个样本不同 seed）
- VMAD：`seed = cfg.seed`（固定 seed=42）

当前实验因均为 SELF 复现模式（同一 sample 对比自己），seed 策略差异不影响结果。但多样本评测时需注意一致性。
