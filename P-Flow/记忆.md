## 一、项目背景

- **模型**: Wan2.1-1.3B T2V（文生视频）
- **编码器特性**: UMT5 encoder 存在位置偏置（U 形分布）：position 0 和最后一个 token 的权重是其他位置的 10-15 倍
- **评测指标**: CLIP（帧级视觉相似度）、XCLIP（时序一致性）
- **VLM**: Qwen2.5-VL-7B（负责"看"视频差异）
- **LLM**: qwen-plus（负责改写 prompt）

---

## 二、核心实验发现

| 策略 | CLIP | XCLIP | 结论 |
|------|------|-------|------|
| Position 优化（主体前置） | +0.0071 | -0.0032 | 帮 CLIP 但伤 XCLIP |
| Structured 改写（时序链） | -0.0023 | +0.0089 | 帮 XCLIP 但伤 CLIP |
| **Fusion 融合策略** | **+0.0106** | **+0.0143** | **唯一同时提升两者的方法** |

融合策略的本质：在原始 VLM prompt 上做**最小侵入式修改**——只改动作时态和首尾位置，其余一字不动。

---

## 三、改写三原则（Fusion Strategy）

### 1. 开头：Subject-First Opening
- 去掉 "The video captures/depicts/showcases..." 框架词
- 直接以核心动作主体名词开头（如 "Two small sailboats"、"White SUV"、"Giant whale"）
- 利用 position-0 的黄金权重

### 2. 动作：Temporal Action Chain
- 将静态描述改为时序链（"is running" → "initially trots, then sprints, finally slows"）
- 只在动作主体的运动上加 1-2 个时间标记词（initially/then/gradually）
- 绝不在背景、光线、氛围上加时间标记
- 这是 XCLIP 提升的关键

### 3. 结尾：Key Visual/Motion Feature
- 用最关键的运动/镜头特征词收尾（如 "gentle circular motion"、"dust trail billowing"）
- 利用最后一个 token 的黄金权重

### 4. 中间：Preserve Visual Vocabulary
- 保留 VLM 原始的视觉描述词不动（颜色、材质、具体物体名称、空间关系）
- 这些是 CLIP 得分的基础
- 不允许同义替换（"dark brown hulls" ≠ "wooden hulls"，"glass facades" ≠ "glass walls"）

## 四、Pipeline 架构

```
Baseline VLM Caption
        │
        ▼ (llm_rewrite, REWRITE_SYSTEM)
   iter0 prompt ──→ 生成视频 ──→ 评测 CLIP/XCLIP
        │                           │
        │    ┌──────────────────────┘
        │    ▼
        │  VLM 看 [原始视频 ↕ 生成视频] 拼接 → 输出差异描述
        │    │
        │    ▼ (llm_refine, REFINE_SYSTEM)
        │  LLM 根据差异修复 prompt（只改 top 1-2 差异）
        │    │
        ▼    ▼
   iter1 prompt ──→ 生成视频 ──→ 评测 ──→ VLM对比 ──→ LLM修复 ──→ iter2 ...
```

- **VLM 只"看"**：把原始视频和生成视频上下拼接，分析差异
- **LLM 只"写"**：根据 VLM 的差异反馈，针对性修复 prompt

## 六、REWRITE_SYSTEM Prompt（V4 当前版本）

核心改进（相比 V3）：
- 加入"主体识别"专门章节：6 个正反对比示例（whale vs cityscape, airplanes vs jungle 等）
- 改为"3 surgical changes"框架：强调"复制整个输入，只改 3 处"
- 加入 4 个完整 few-shot examples（含 WHY 解释），比 V3 多了 #7 sailboats 和 #17 SUV
- 明确禁止列表：不压缩、不改措辞、不合并段落
- 加入 Process 步骤：先识别主体 → 复制全文 → 改 3 处

```
You restructure VLM video captions into better T2V generation prompts. Your output must be nearly identical to the input — you only make 3 surgical changes.

## CRITICAL: How to identify the ACTION SUBJECT
The action subject is THE THING THAT MOVES OR ACTS. Ask: "What is performing the main motion?"
- whale swims through cityscape → subject = "Giant whale", NOT "Underwater cityscape"
- paper airplanes fly through jungle → subject = "Colorful paper airplanes", NOT "Vibrant jungle"
- SUV drives on road → subject = "White SUV", NOT "Scenic mountainous landscape"
The subject is NEVER the background/environment/setting.

## The 3 changes (NOTHING ELSE):
1. OPENING — Move action subject to first words. Delete "The video shows/captures..."
2. ACTION — Find 1-2 motion sentences → add temporal chain (initially/then/gradually)
3. ENDING — End last sentence with vivid motion/visual keyword

## What you must NOT do:
- Do NOT compress (150 words in → ~150 words out)
- Do NOT rephrase visual descriptions (copy VERBATIM)
- Do NOT merge paragraphs
- Do NOT add temporal markers to background/lighting sentences

## Examples: 4 个（鲸鱼 #31、纸飞机 #21、帆船 #7、SUV #17），每个含 WHY 解释
```

## 七、llm_rewrite() User Message（V4）

```python
f"Restructure this VLM caption ({word_count} words). "
f"First, identify what MOVES in this caption — that is your action subject. "
f"Then make ONLY 3 changes: "
f"(1) move the action subject to the first word, "
f"(2) find the 1-2 sentences about the subject's motion and add a temporal chain, "
f"(3) end the last sentence with a key motion/visual word. "
f"Copy ALL other sentences VERBATIM — do not rephrase, compress, or merge paragraphs. "
f"Output must be ~{word_count} words (±15%). Do NOT compress."
```

- 加了更多细节、示例、保留原文的约束
- 问题：仍然不够匹配用户手动改写的风格

- 分析了用户 10 个手动改写 vs baseline 的系统性差异
- 加入 3 个真实 few-shot examples（鲸鱼、纸飞机、小狗）
- 明确了"只改 3 个地方，其余一字不动"的核心原则
- 加入了 UMT5 位置偏置的解释

### V4（当前版本）
- 针对 V3 实验暴露的 3 个问题（选错主体、压缩、改措辞）重写

- 明确禁止列表 + Process 步骤

## 十四、V4 Prompt 核心变化（对比 V3）

### 问题诊断（V3 的 3 个致命问题）
1. **选错主体**：LLM 把场景/环境当主体（如 #31 选了 "Underwater cityscape" 而非 "Giant whale"）
2. **内容压缩**：150 词输入被压缩到 50-60 词，丢失大量视觉细节
3. **改写措辞**：把原文的视觉描述换成同义词，破坏了 UMT5 的 token 匹配

### V4 修复策略
1. **新增主体识别区**：6 组正确/错误对比示例，明确"主体 = 画面中移动的实体"
2. **新增禁止事项**：5 条 "What you must NOT do"（不删句、不合并句、不换词、不缩短、不改视觉描述）
3. **改变框架**：从"重写 caption"变为"复制全文，只做 3 处手术式修改（开头/动作/结尾）"
4. **4 个长示例**：每个都有 WHY 注释解释为什么这样改（#31 whale, #21 airplanes, #7 sailboats, #17 SUV）
5. **user message 也更新**：强调 "identify what MOVES" 和 "copy ALL other sentences VERBATIM"

## 十五、V5 工程改进（prompt 之外的优化）

### 改动清单

| # | 改动 | 文件 | 目的 |
|---|------|------|------|
| 1 | temperature 降低 | run_hybrid_iter.py | REWRITE 0.5→重试降到 0.3, REFINE 0.4→重试降到 0.2 |
| 2 | length check + 自动重试 | run_hybrid_iter.py `llm_rewrite()` | 输出 < 70% 输入字数时自动重试（最多 2 次） |
| 3 | diff check + 自动重试 | run_hybrid_iter.py `llm_rewrite()` | 编辑距离 > 50% 时自动重试 |
| 4 | REFINE 也加 length/diff check | run_hybrid_iter.py `llm_refine()` | 修复阶段：length < 70% 或 edit > 35% 时重试 |
| 5 | REFINE_SYSTEM 大幅强化 | run_hybrid_iter.py | 加禁止事项、4 类常见反馈的修复指引、1 个 few-shot 示例 |
| 6 | VLM 结构化对比 prompt | run_hybrid_iter.py `VLM_COMPARE_PROMPT` | 4 维度对比（SUBJECT/MOTION/BACKGROUND/TIMING），直接调 VLM._generate() |
| 7 | `_compute_edit_ratio()` 工具函数 | run_hybrid_iter.py | 基于 SequenceMatcher 的 token-level 编辑距离 |

### 分阶段验证方案

新增脚本 `scripts/verify_rewrite_quality.py`，支持 3 种消融模式：

```bash
# Phase 1: 全部改动（V4 prompt + temp 0.5 + length/diff check）
python scripts/verify_rewrite_quality.py --ablation base --baseline_dir ... --output_dir outputs/verify_base

# Phase 2: 去掉 check（V4 prompt + temp 0.5，无验证重试）
python scripts/verify_rewrite_quality.py --ablation no_check --baseline_dir ... --output_dir outputs/verify_no_check

# Phase 3: 高温度（V4 prompt + temp 0.7，无验证）
python scripts/verify_rewrite_quality.py --ablation high_temp --baseline_dir ... --output_dir outputs/verify_high_temp
```

验证指标：
- **Length Ratio**：输出/输入字数比（目标 85-115%）
- **Edit Distance**：与输入的编辑距离（目标 20-40%，太低=没改，太高=乱改）
- **Subject Accuracy**：开头是否包含正确主体（目标 100%）
- **GT Similarity**：与手动 ground truth 的相似度（目标 >60%）

### 验证逻辑

```
Phase 1 vs Phase 2 → length/diff check 是否有效？
  - 如果 Phase 1 的 Length Ratio 更接近 100%，说明 check 有效
  - 如果 Phase 1 的 Subject Accuracy 相同，说明 check 不影响主体识别

Phase 2 vs Phase 3 → temperature 降低是否有效？
  - 如果 Phase 2 的 Edit Distance 更低，说明低温度减少了不必要改动
  - 如果 Phase 2 的 GT Similarity 更高，说明低温度更贴近 ground truth

Phase 1 vs GT → V4 prompt 整体效果
  - GT Similarity > 60% 说明自动改写接近手动水平
  - Subject Accuracy = 100% 说明主体识别问题已解决
```

### 完整实验流程（推荐）
