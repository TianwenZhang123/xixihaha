### 2.3 质量门控机制回顾

当前 FI 门控流程（代码 `pipeline.py:1295-1320`）：

```
λ_max (用户设定, 如 0.05/0.10)
  ↓
schedule(t) → λ_sched (如 middle_peak: 中间高两端低)
  ↓
× quality_scale (基于 η_temporal 帧间余弦相似度, 0.1~1.0)
  ↓                    ┌──────────────────────────────┐
  ↓                    │ mean_cos > threshold → 高scale │
  ↓                    │ mean_cos < threshold → 低scale │
  ↓                    └──────────────────────────────┘
  ↓ = λ_eff
  ↓
× adaptive_gate (基于每步 cos_sim(h_gen, h_ref), sigmoid 门控)
  ↓
= λ_actual (最终实际注入量)
```

**关键点**: quality_scale **不会完全阻止注入**（除非 < 1e-6），只是降低名义 λ。adaptive_gate 会进一步逐层调节。

## 三、门控深度分析（2026-06-25）

> **核心发现**: 当前两套门控系统都有严重问题 — SVD BidiStdGate 根本没启用，FI adaptive_gate 把注入压到了 31%。

### 3.1 SVD BidiStdGate — 根本没启用！

**日志证据**:

```
搜索 "strategy=bidi_std_gate" → 0 匹配!
搜索 "BidiStdGate FLOOR"      → 0 匹配!
搜索 "BidiStdGate CAP"         → 0 匹配!
```

**代码原因** (`pipeline.py:664-673`):

BidiStdGate 逻辑位于 `_compute_pna_score()` 函数内，该函数只有在**同时满足两个条件**时才会被调用：

```python
# pipeline.py:664-673
if cfg.adaptive_alpha:                    # ← 条件1: 需要 --adaptive_alpha
    if getattr(cfg, 'pna_probe', False):  # ← 条件2: 需要 --pna_probe
        alpha_eff = self._compute_pna_score(...)  # BidiStdGate 在这里面!
    else:
        # TSR-guided 简单路径 (无 BidiStdGate)
else:
    alpha = cfg.alpha  # 固定 α (无任何门控)
```

之前所有命令都**缺少** `--adaptive_alpha` 和 `--pna_probe`，导致代码直接走了 `alpha = cfg.alpha` 的固定路径。

虽然 `pna_std_gate` 默认为 `True`（`run.py:196` 中 `pna_std_gate=not args.no_pna_std_gate`），但它只在 `_compute_pna_score()` 内部生效，而该函数从未被调用过。

**这是一个代码架构设计问题**: `--pna_std_gate` 看起来是独立开关，实际上依赖 `--pna_probe` 才能生效。

**PNA v5 的 30-case 实际上跑的是 α=0.004 固定值，没有任何门控介入！**

### 3.3 FI 门控链 — adaptive_gate 把注入压到了 31%！

FI 的实际注入量经过三层压缩：

```
λ_max (用户设定, 如 0.05/0.10)
  ↓
× quality_scale (基于帧间余弦相似度)
  ↓   实测: min=0.196, max=0.966, mean=0.550
  ↓   → 平均只保留名义量的 55%
  ↓
× adaptive_gate (基于每步 cos_sim(h_cur, h_ref), sigmoid门控)
  ↓   实测: min=0.206, max=0.471, mean=0.313
  ↓   → 平均只保留 31% 的 scheduled lambda
  ↓
= λ_actual (最终实际注入量)
  → 平均只有名义 λ 的 55% × 31% = **17%** 真正被注入!
```

**实测 adaptive_gate 数据**:

```

cos_sim: min=0.523, max=0.770, mean=0.660
=> 平均只注入名义 lambda 的 31.3%
```

**方案**: 提高 threshold 到 0.00，降低 k 到 10，提高 min_scale 到 0.3

#### 方向 C：降低 FI adaptive_gate 温度（特征对齐门控）

## 九、关键发现总结

1. **eta_std 呈 U 型分布**：低和高都不好，中间甜点区最佳
2. **v4 的 α=0.0004 严重不足**：比 baseline 差 0.0277

4. **需要双向保护**：既防欠注入（floor），也防过注入（cap）
5. **FI λ 可以从 0.05 提高到 0.10** — Adaptive gate 会自动调节实际注入量，不会过注
6. **Adaptive gate 必须开启?** ⚠️ **存疑!** — 之前 no_adaptive 降分(0.84→0.82)，但那是因为 λ=0.05+无门控=过注; 如果降低λ可能反转

8. **PASS/REJECT 标注不可靠** — REJECT 组 4/5 实际从 FI 受益，quality_scale 与效果相关性弱
9. **Case 31 是唯一真正"抗注入"的 case** — quality_scale 不低(0.527)但 XCLIP -15.3%，可能是场景特殊性
10. **[★★★]** **SVD 门控策略确定**: v5 fixed α=0.004 + 单向 Floor 保护(η_std<0.32 → α→0.006), 无需 --adaptive_alpha/--pna_probe
