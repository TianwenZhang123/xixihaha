## α 网格搜索实验

**时间**：2025-06-04

**目的**：上次 SVD 实验使用 α=0.001（极保守），本次搜索 α ∈ {0.001~0.005}，寻找 CLIP 与 XCLIP 的最优平衡点。

### 背景

噪声混合公式：`η = √α · η_temporal + √(1-α) · η_random`。α 越大，运动先验信号越强。

### 实验设置

- 搜索空间：α ∈ {0.001, 0.002, 0.003, 0.004, 0.005}
- Prompt：V4 iter1 最佳 prompt（`hybrid_iter_v4/captions_iter0`）
- SVD 参数：ρ_s=0.1, ρ_m=0.9
- 生成参数：steps=30, guidance=5.0, seed=42, 480x832, 81帧
- 样本：7, 17, 21, 31, 32, 33, 34, 43, 46, 47（共 10 个）

### 运行命令

```bash
cd /root/autodl-tmp/videofake/P-Flow && python scripts/svd_grid_search.py \
    --data_dir /root/autodl-tmp/data/video-200/water_mark_out \
    --caption_dir /root/autodl-tmp/outputs/hybrid_iter_v4/captions_iter0 \
    --output_dir /root/autodl-tmp/outputs/alpha_search
```

α=0.004-0.005 是最优区间。α=0.004 实现最佳综合表现（CLIP 0.8953, XCLIP 0.7667），颠覆了此前 α=0.001 的保守假设。更强的运动先验（√α 从 0.032 增至 0.063）不会过度约束模型，反而显著提升了生成质量。

## Velocity Matching 首次验证

**时间**：2025-06-03

**目的**：验证修复 `max_sequence_length` bug（226→512 对齐）后，Velocity Matching 能否在 noise prior 基础上带来增益。

### Bug 修复

**问题**：`_encode_prompt()` 调用 `pipe.encode_prompt()` 时未显式传入 `max_sequence_length`，默认值 226 与生成阶段的 512 不匹配。导致 Δe 在 226-token 空间优化后注入 512-token 空间，后 286 位全零，运动信息严重丢失。

**修复**：统一为 `max_sequence_length=512`。

### 实验设置

配置：`--velocity_full`（inversion + svd + blend + velocity），α=0.004，es=0.005，velocity_steps=30，lr=0.001。

### 结果

| 方案 | CLIP | XCLIP | 说明 |
|------|------|-------|------|
| noise prior only (α=0.004) | 0.8953 | 0.7667 | baseline |
| velocity es=0.005 | 0.8990 | 0.7639 | CLIP 微升，XCLIP 微降 |

**结论**：es=0.005 注入量 ~0.04 太弱，信号被淹没。需要提高 embed_strength。

## embed_strength 推导与网格搜索

**时间**：2025-06-04

**目的**：基于 VMAD 精细参数扫描，推导 P-Flow 中 es 的理论最优值。

### 推导过程

注入公式：`e_final = e0 + es * Δe`，模型感知信号大小 = `||es * Δe||`。

VMAD 验证的最优注入量区间：||injection|| = 0.18~0.29。

| VMAD es | ||Δe|| (VMAD) | ||injection|| | 效果 |
|---------|--------------|---------------|------|
| 0.005 | 36.25 | 0.18 | CLIP 最优 |
| 0.008 | 36.25 | 0.29 | XCLIP 最优 |
| 0.01 | 36.25 | 0.36 | 略过强 |

P-Flow 的 ||Δe||~8.5（30步无 position-aware，仅 VMAD 的 1/4）。等效映射：

- CLIP 最优：es = 0.18 / 8.5 = 0.02
- XCLIP 最优：es = 0.29 / 8.5 = 0.034

### 网格搜索结果

```bash
for es in 0.02 0.05; do
    python run.py ... --velocity_full --alpha 0.004 --embed_strength $es ...
done
```

| 方案 | CLIP | XCLIP | ||es*Δe|| | 说明 |
|------|------|-------|-----------|------|
| noise prior only | 0.8953 | 0.7667 | 0 | baseline |
| velocity es=0.005 | 0.8990 | 0.7639 | ~0.04 | 太弱 |
| **velocity es=0.02** | **0.8981** | **0.7705** | **~0.17** | 最优 |
| velocity es=0.05 | 0.8965 | 0.7685 | ~0.43 | 略过强 |

**结论**：es=0.02 最优，XCLIP 首次超过 baseline（+0.38%），实际注入量 0.17 恰好落在 VMAD 最优区间下界，与理论一致。

## 60步实验与失败分析

**时间**：2025-06-03

**目的**：验证增加 velocity matching 优化步数（30→60）能否进一步提升。

### 结果

| 配置 | ||Δe|| avg | 注入量 (es*||Δe||) | CLIP | XCLIP |
|------|-----------|-------------------|------|-------|
| 30步, es=0.02 | ~8.5 | 0.17 | **0.8981** | **0.7705** |
| 60步, es=0.02 | ~12.5 | 0.25 | 0.8935 | 0.7628 |

### 失败原因

60步指标全面下降，三个根因：

1. **注入过强**：60步使 ||Δe|| 增大到 12.5，注入量 0.25 超出最优区间 0.17，落入"过强"区域。

2. **优化不稳定**：v1 每步只采 1 个随机时间步，梯度方差大。更多步数中方差累积，部分样本（31, 47）出现 loss 振荡（final_loss > 0.5），Δe 方向在不同时间步之间拉锯。

3. **能量分散**：全部 512 token 位置（含大量 padding）被均匀优化，有效 token 获得的梯度能量不足。

### 关键启示

问题不是"步数不够"，而是"每步优化效率低"。需要从代码层面改进：降低梯度方差、集中有效能量、关注运动区域。→ 催生 v2 三项改进。

## Velocity Matching v2 改进设计与验证

**时间**：2025-06-05

**目的**：基于60步实验暴露的三个根因，在不增加步数的前提下从代码层面提升优化效率。

### 设计背景

v1 核心循环：每步采 1 个随机 t → 全部 512 token 做 forward → 均匀 MSE → 更新 Δe。梯度信号的"信噪比"低，被三种噪声源污染：时间步采样随机性、padding 位置的无效梯度、静态区域的无用优化。v2 针对每个噪声源设计一项改进。

### 改进 1：分层多时间步采样（Stratified Multi-Timestep Sampling）

**论文基础**：CVPR 2025 "Adaptive Non-Uniform Timestep Sampling" 证明扩散模型训练中梯度方差跨时间步差异巨大（最大/最小方差比可达 100x），分层采样可显著加速收敛。NVIDIA CARV (2025) 的 stratified-inverse-CDF 额外带来 ~25% 方差下降。

**v1 问题**：每步只采 1 个随机 t，30步仅覆盖 30 个时间点。对运动复杂视频，不同 t 的 velocity 差异大，单点采样梯度方向随机性高。

**v2 方案**：将 [0, T_m] 分成 K=4 个等宽 bin，每 bin 内均匀采一个 t，每步 4 次 forward+backward 后平均梯度。覆盖密度从 30 提升到 120 点，方差下降 sqrt(K)=2 倍。

**实现**：`VelocityMatcher._sample_stratified_timesteps(K, dtype)`

**显存影响**：几乎为零（循环 4 次，不是 batch 4 份）。

### 改进 2：Padding-Aware 梯度 Mask

**论文基础**：NAACL 2025 "Padding Tone" 分析 padding tokens 在 T2I cross-attention 中的作用。我们对 Wan2.1 DiT 的 attention dump 证实：padding 区域（pos 154~263）有 4-5x 异常高权重，模型在"看"这些无语义位置。

**v1 问题**：全部 512 位置统一优化，但实际 caption 只占 120-310 token。能量被分散到 padding。

**v2 方案**：tokenize caption 获取有效 token 长度 → 构建 padding_mask → 每步 backward 后 `delta_e.grad.mul_(padding_mask)` 将 padding 梯度置零。

**实现**：`pipeline._get_token_length(caption)` + `VelocityMatcher.optimize(token_length=...)`

### 改进 3：运动感知 Loss 加权（Motion-Aware Weighting via LTD）

**论文基础**："Latent Temporal Discrepancy as Motion Prior" (2025) 提出帧间 latent 差异作为运动先验指导 loss 加权。Motion Inversion (SIGGRAPH 2025) 通过 frame-to-frame debiasing 确保 embedding 编码运动而非静态外观。

**v1 问题**：均匀 MSE 对所有像素同等权重。但视频复现核心是运动——静态背景即使 v_pred 有偏差也不影响感知质量。

**v2 方案**：计算 v* 的帧间方差作为 per-pixel motion weight，高运动区域获得更高 loss 权重。

**公式**：
```
sigma2(x,y) = Var_f(v*[:,:,f,x,y])
w(x,y) = 1 + lambda * log(1 + sigma2(x,y) / mu)
w_normalized = w / mean(w)
L = mean(w * (v_pred - v*)^2)
```

**实现**：`VelocityMatcher._compute_motion_weight(v_star)`，参数 `motion_weight_strength`。

### 代码改动汇总

| 文件 | 改动内容 |
|------|---------|
| `src/velocity_matching.py` | 整类重写：新增 `_sample_stratified_timesteps()`、`_compute_motion_weight()`；优化循环内 K 次 forward+backward |
| `src/pipeline.py` | PFlowConfig 新增 `velocity_K`、`velocity_motion_weight`；新增 `_get_token_length()` 方法 |
| `run.py` | 新增 `--velocity_K` 和 `--velocity_motion_weight` CLI 参数 |

### 全量结果分析

**结论：v2 全量提升有限（CLIP +0.2%, XCLIP +0.4%），远不如 quick 验证的 +4.9%。**

表现呈两极分化：

- **明显受益的样本**：31（XCLIP +5.4%）、43（+1.5%）、32（+1.0%）——高运动 or 中等偏低表现的样本
- **退化的样本**：21（XCLIP -1.7%）、47（-1.8%）、7（-1.0%）——运动相对温和的样本

**退化原因分析**：

1. **Motion weight 对低运动视频有害**：样本 7（帆船漂浮，motion weight max=2.375 但整体方差低）、47（两只狗散步，运动温和）的 v* 帧间方差本身就小，motion weight 把 loss 权重给了微小抖动区域（可能是噪声而非有意义运动），导致 Δe 方向偏移。

2. **样本 21 仍有收敛问题**：v1 中该样本就出现 loss 反弹（final_loss=0.332 > step0=0.099），v2 的 stratified sampling 改善了覆盖但没根治不稳定性。

3. **Quick 验证选样偏差**：quick 选的 4 个样本恰好都是 v2 擅长的类型（高运动/短 prompt），导致估计偏乐观。

**与 quick 验证数据差异说明**：quick 使用的 v1 baseline 来自 `es_grid/es_0.02`（symlink 扁平化），全量 v1 来自 `velocity_fix_512`（直接生成），两次 v1 的评测数值存在微小差异，但不影响 v2 的绝对评测结果。

### 下一步改进方向

1. **Adaptive motion weight**：当 v* 帧间方差低于阈值时自动降低 motion_weight_strength（避免在低运动视频上放大噪声）
2. **降低 motion_weight_strength**：从 1.0 调到 0.3~0.5，温和化加权
3. **es 联合调参**：v2 的 padding mask 使有效 token 上的 ||Δe|| 更集中，最优 es 可能需要微调
4. **逐项消融**：确认三项改进各自的独立贡献，找出哪项导致退化

## 附录：Velocity Matching 原理

**Δe 优化流程**：

1. 参考视频 → VAE encode → `z0`
2. 参考视频 → Flow Inversion (50步 ODE 反推) → `eta_inv`
3. 目标速度场：`v* = z0 - eta_inv`（rectified flow 解析解）
4. 30 步梯度下降，每步：采样 t → `x_t = (1-t)*eta_inv + t*z0` → `v_pred = DiT(x_t, t, e0+Δe)` → Loss = `||v_pred - v*||^2` → 只更新 Δe（模型冻结）
5. 生成时注入：`e_final = e0 + es * Δe`

**为什么有效**：Flow Matching 训练目标是 `v_theta(x_t, t, c) = z0 - eta`。微调条件 c 从 e0 到 `e0+Δe`，等于告诉模型"把 ODE 终点精确指向参考视频的 latent"，Δe 编码了文字无法表达的运动细节。
