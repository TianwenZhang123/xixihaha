## 一、Wan2.1 模型输入与 P-Flow 可操作接口

### 1.1 Wan2.1 Pipeline 接受的完整输入

P-Flow 使用 diffusers 官方 `WanPipeline`，通过标准 `__call__` 接口驱动生成。根据代码（`src/pipeline.py:1568-1582`），实际传入的参数如下：

| 参数 | 类型 | 官方默认值 | P-Flow 使用值 | 说明 |
|------|------|--------|--------|------|
| `prompt` | str | None | VLM caption | 文本描述，经内部 UMT5 text encoder 编码为 token embeddings，作为 cross-attention 条件 |
| `negative_prompt` | str | None | 硬编码 NEGATIVE_PROMPT | 负面提示词，用于 CFG 引导 |
| `height` | int | 480 | 480 | 视频高度（像素） |
| `width` | int | 832 | 832 | 视频宽度（像素） |
| `num_frames` | int | 81 | 81 | 生成帧数（VAE 压缩后时间维 F=21） |
| `guidance_scale` | float | 5.0 | 5.0 | Classifier-Free Guidance 强度 |
| `num_inference_steps` | int | **50** | **30** | Flow Matching 去噪步数（P-Flow 自行设为 30） |
| `generator` | torch.Generator | None | seed=42 | 随机数生成器（控制 η_random） |
| `output_type` | str | **"np"** | **"pt"** | 输出格式（官方默认 numpy，P-Flow 用 PyTorch tensor） |
| `latents` | Tensor / None | None | SVD z_T / None | **初始噪声 z_T**，形状 `[1, 16, F, H, W]`。若为 None 则内部随机采样 N(0,1) |
| `callback_on_step_end` | Callable / None | None | 灰盒回调 / None | 每步去噪后的回调函数（灰盒操作入口） |
| `callback_on_step_end_tensor_inputs` | list | ["latents"] | ["latents"] | 回调可访问的 tensor 名称 |

> **验证结论（对照 [huggingface/diffusers main 分支](https://github.com/huggingface/diffusers/blob/main/src/diffusers/pipelines/wan/pipeline_wan.py)）**：上述 12 个参数**全部是 WanPipeline 官方 API 原生支持的参数**，P-Flow 没有新增任何自定义参数。官方 `__call__` 共有 18 个参数，P-Flow 未使用的 6 个为：`guidance_scale_2`（二阶段 transformer 专用）、`num_videos_per_prompt`、`prompt_embeds`、`negative_prompt_embeds`、`return_dict`、`attention_kwargs`、`max_sequence_length`。P-Flow 仅在 `num_inference_steps`（50→30）和 `output_type`（"np"→"pt"）上使用了非默认值。

### 1.2 P-Flow 的三个操作接口

根据代码逻辑，P-Flow 将 Wan2.1 作为几乎不改动的黑盒/灰盒使用，真正可操作的接口有三个：

**接口 1：prompt（文本条件）**—— L1 层操作对象。prompt 以字符串形式传入，由 WanPipeline 内部调用 UMT5 编码。P-Flow 不直接传 `prompt_embeds`（`encode_prompt` 仅在 inversion 阶段单独使用）。

**接口 2：latents（初始噪声）**—— L2 层操作对象。通过 inversion + SVD 滤波 + blend 构造结构化初始噪声，替代默认的纯随机 N(0,1)。这是纯黑盒下唯一能传递参考视频信息的通道。

**接口 3：callback_on_step_end（去噪过程钩子）**—— 灰盒操作入口。利用 diffusers 的回调机制，在每步 denoise 后拿到当前 latents 并施加外部约束（如轨迹锚定的 lerp），等价于在 Flow ODE 上加弹性恢复力。

此外，inversion 阶段会绕过 `__call__` 直接调用 `pipe.transformer`（前向传播计算速度场）和 `pipe.vae`（视频编码/解码），但这属于预处理阶段，不影响生成 pipeline 的行为。

## 二、SVD Noise Prior 的设计动机与作用

### 2.1 P-Flow 论文中 SVD 的核心作用

P-Flow（论文 Section 3.3）明确指出，SVD noise prior 的**核心作用不是"引导运动方向"，而是"稳定迭代 prompt 优化的收敛"**：

- **问题**：P-Flow 的核心 pipeline 是迭代 prompt 优化——每轮用 VLM 评估生成结果与参考视频的差异，据此改写 prompt，再重新生成。但如果每轮生成使用完全随机的 z_T，则视觉效果的变化来源混杂了"prompt 变化"和"噪声随机性"两个因素，优化器无法区分，收敛困难。
- **解决**：通过 inversion + SVD 提取参考视频的时序结构作为确定性分量 η_temporal，以极小比例（α=0.001~0.004）blend 进 z_T。这提供了一个**跨迭代的稳定锚点**——每轮生成的 z_T 都包含相同的微弱确定性偏置，使生成结果的变化**主要归因于 prompt 变化而非噪声随机性**，从而稳定优化方向。
- **平衡**：`√(1-α) ≈ 0.998`，z_T 仍然 99.8% 是随机的，保留了足够的探索多样性。如果 α 过大（如固定噪声），则陷入局部最优无法探索。

### 2.2 我们实验中 SVD 的实际效果

在我们的实验设定中（非迭代优化，单次生成），SVD v1 的效果体现为：η_temporal 携带的参考视频时序结构作为 z_T 的微弱方向偏置，被 30 步 flow matching 积分过程累积放大，最终使生成视频在运动模式上与参考视频更相似。实测 CLIP +2.4%, XCLIP +5.1%（vs 纯随机噪声 baseline）。

关键发现（来自 10+ 组消融实验）：T2V 模型对 z_T 中的结构化偏离极度敏感。所有试图增强注入的改进均失败，注入量越大效果越差。α=0.004（实际方向偏移 ~2%）是纯黑盒下的绝对天花板。

## 三、当前有效策略总结

### 3.1 纯黑盒最优：原始 Caption + SVD v1

- **L1（Prompt）**：保留 VLM 原始 caption 不改写，运动描述模糊（"the boat moves"），给 SVD 留出信息补充空间
- **L2（SVD）**：两阶段 SVD 滤波（空间去内容 + 时间保运动），v1 模式不做 renorm，α=0.004 极小剂量 blend
- **结果**：CLIP 0.8964, XCLIP 0.7874（当前黑盒最优）

### 3.2 L1 有效改进：v9 删减+VLM补充

- **策略**：LLM 对原始 caption 做纯删减（去 preamble、hedging、summary），再用本地 Qwen2.5-VL-7B 看视频帧补充视觉细节，**不碰运动描述**
- **与 SVD 兼容**：不描述运动 → 不与 SVD 偏置冲突
- **结果**：XCLIP +1.3%，CLIP 持平

### 3.3 灰盒方向：Latent Trajectory Soft Anchor

- **原理**：参考视频反演得到 latent 轨迹 {z_0,...,z_T}，生成时每步 denoise 后 lerp 向参考轨迹：`z_anchored = (1-β_t)*z_gen + β_t*z_ref`
- **调度**：`warmup_decay`（前 20% step 从 0→β_max，再余弦衰减到 0），避免早期干扰
- **质量门控**：trajectory coherence < 0.3 的混沌样本自动跳过 anchor
- **状态**：β=0.1 + warmup_decay + quality gate 实验中

## 五、关键发现与结论

### 5.1 SVD 的双重角色

在 P-Flow 原论文的迭代优化 pipeline 中，SVD 的核心作用是**提供跨迭代的确定性锚点**（stabilize optimization convergence）。在我们的单次生成实验中，SVD 的作用退化为**提供运动方向的微弱统计偏置**。两种作用都要求 α 极小——前者是为了保留探索多样性，后者是因为 T2V 模型无法容忍 z_T 的结构化偏离。

### 5.2 L1 与 L2 的结构性矛盾

SVD 先验本质是"运动信息补充器"——当 prompt 说不清怎么动时有效（如原始 VLM caption）。当 prompt 已精确描述运动（v7e），SVD 偏置变为干扰。正确分工：L1 只做结构优化不碰运动，L2 负责运动引导。

### 5.3 纯黑盒天花板

10+ 组消融实验证明：η_temporal 的一切信息（内容、幅度、相位）在强注入时对 T2V 模型均有害。α=0.004 是绝对天花板——信号弱到"不产生破坏"是其有效的前提条件。

### 5.4 灰盒方向是突破口

轨迹锚定绕过"只能改 z_T"的黑盒限制，通过 `callback_on_step_end` 在整个 ODE 求解过程中施加软约束。数学上等价于在 Flow ODE 上加轨迹正则项——这与 P-Flow 原文的"仅影响起点"形成本质区别（我们影响整个生成路径），构成独立贡献。

## 六、SVD v1 详细流程

```
参考视频 V_ref
    │
    ▼
[VAE Encode] → z_0, shape [1, 16, 21, 60, 104]
    │
    ▼
[Flow Matching Inversion] → η_inv (反演噪声, t=0→t=1, Euler 30 steps)
    │   调用 pipe.transformer + pipe.vae，计算 ODE 积分的逆过程
    ▼
[SVD Stage 1: 空间去内容]
    │   reshape (C×F, H×W) → SVD → 去除 top-k 空间主成分(占 90% 能量)
    ▼
[SVD Stage 2: 时间保运动]
    │   reshape (C×H×W, F) → SVD → 只保留 top-k 时间主成分(占 90% 能量)
    │   输出 η_temporal, std ≈ 0.28~0.41
    ▼
[Noise Blend] → z_T = √α·η_temporal + √(1-α)·η_random
    │   α=0.004, 实际方向偏移 ≈ 0.063 × 0.3 ≈ 2%
    ▼
[WanPipeline.__call__(prompt, latents=z_T, ...)]
    │   30 步 flow matching 去噪
    ▼
生成视频
```

## 附录 B：L2+L3 融合失败分析与方案设计

> **分析时间**：2025-06-12
>
> **背景**：Pure L2（SVD v1, α=0.004）和 Pure L3（Trajectory Anchor V3 gated）各自有效，但朴素组合导致严重退化。本附录深入分析失败根因并设计修复方案。

### B.1 各独立实验的 Baseline 数据

| 实验配置 | CLIP | XCLIP | 说明 |
|---------|------|-------|------|
| 纯 Baseline（无优化，裸 VLM caption） | 0.8753 | 0.7491 | 6-11周会 #0 |
| Pure L2（SVD v1 α=0.004，裸 caption） | **0.8964** | **0.7874** | 6-11周会 #1，黑盒最优 |
| Pure L3（Anchor V3 + warmup_decay + quality gate） | **0.8845** | **0.7810** | 6-11周会 #22 |
| L2+L3 方案B（adaptive cos threshold=0.2） | ~0.770 | ~0.604 | 严重退化 |

### B.2 当前代码数据流（L2+L3 联合模式）

```
参考视频 V_ref
    │
    ├──[Step 3: _compute_noise_prior]──────────────────────────────
    │   VAE encode → z_0 (ref_latents_enc)
    │   Flow Inversion (Euler 50步, t=1→t=0) → η_inv (raw, std≈1.0)
    │   SVD 两阶段滤波 → η_temporal (去内容保运动, std≈0.3)
    │
    ├──[Step 3.6: invert_with_trajectory]──────────────────────────
    │   复用 ref_latents_enc + prompt_embeds
    │   Flow Inversion (Euler 50步, t=1→t=0, 缓存每步中间态)
    │   → ref_trajectory: {t=1.0: z_0, t=0.98: ..., ..., t=0.0: η_inv}
    │   ⚠️ 与 Step 3 做的是同一个确定性 ODE，轨迹终点 = η_inv (raw)
    │
    ├──[Step 4: 生成]──────────────────────────────────────────────
    │   L2: _get_latents(η_temporal, generator)
    │       → z_T = √α·η_temporal + √(1-α)·η_random, α=0.004
    │       → z_T ≈ 99.6% random + 0.4% SVD-filtered
    │
    │   L3: _generate_with_anchor(prompt, z_T, generator, ref_trajectory)
    │       每步 callback: z_anchored = (1-β_t)*z_gen + β_t*z_ref_t
    │       z_ref_t 来自 ref_trajectory (raw inversion 轨迹上的点)
    │
    └── 生成视频
```

### B.3 核心矛盾：起点不对齐

**Pure L3 为什么成功：**

纯 L3 模式下 `latents=None`，diffusers 内部随机采样 z_T ~ N(0,1)。但同时 `ref_trajectory[t=0.0] = η_inv`，它也是从 z_0 经 ODE 积分到 t=0 的结果。虽然两者不同，但纯 L3 的 quality gate 保证了只有"运动一致"的样本通过，且 warmup_decay 调度（前 20% 从 0.5β 升到 β_max）给了模型足够时间自然靠近。实测纯 L3 的通过样本前期 cos≈0.08，但因为 β_max=0.1 × warmup=0.5 → 实际前 6 步 β≈0.05，shift≈70-80，尚在可接受范围。

**L2+L3 为什么失败：**

L2 模式下 z_T 被 SVD blend 改变：`z_T = √0.004·η_temporal(std≈0.3) + √0.996·η_random(std≈1.0)`。数学上：

```
实际方向偏移 ≈ √α × (η_temporal_std / η_random_std)
            ≈ 0.063 × 0.3 = 0.019 (约 2%)
```

而 `ref_trajectory[t=0.0] = η_inv`（100% 由参考视频决定）。两者 cos 相似度 ≈ 0.07，在 ~500万维空间中几乎正交。

**方案B（cos 门控 threshold=0.2）的失败逻辑：**

1. 前 5-7 步：cos < 0.2，anchor 被跳过 → 等同纯 L2，没问题
2. Step 6-8：cos 刚过 0.2 阈值，anchor 启用
3. 但此时 β_max=0.1（warmup 已到最大），shift=100-140/step
4. 模型每步正常去噪的变化量只有 ~20-30
5. anchor 的 shift 是模型自身变化的 4-5 倍 → 暴力拉拽
6. 结果：生成的视频既不像 prompt 描述的，也无法完全跟踪参考轨迹

**根本问题**：不是"什么时候开始"的问题，而是"力度和对齐度不匹配"的问题。cos=0.2 意味着向量方向差 ~78°，此时施加 β=0.1 的 lerp 等于把 10% 的权重给到一个"几乎正交"的方向。

### B.4 方案设计

> **注：方案 2（cos-proportional β）已于 2026-06-15 实验验证后废弃。** 根本原因：SVD-blended z_T 与 inversion 轨迹不在同一路径上（起点 cos≈0.007，严重不对齐），cos-proportional 仅将 β_max=0.03 的有效强度压缩到 0.002-0.004，轨迹位置 lerp 完全无效。已从代码和命令行参数中删除。

#### 方案 3（学术潜力最大，当前推进方向）：Velocity Direction Anchor / Feature Injection

核心思想：不做位置 lerp，改做方向/速度参考。

当前方案（位置锚定）：
```python
z_anchored = (1-β)*z_gen + β*z_ref_t  # 强制拉到 ref 的"位置"
```

新方案（速度锚定）：
```python
# 从 ref_trajectory 中取相邻两点算出参考"去噪速度"
v_ref = (ref_trajectory[t_prev] - ref_trajectory[t_current]) / dt_ref

# 用参考速度微调生成方向（不关心位置对齐，只关心方向）
z_anchored = z_gen + β * v_ref * dt_gen
```

优势：

- **不要求起点对齐**——不关心 z_gen 在哪里，只关心"下一步往哪走"
- 参考轨迹的速度携带了运动语义（每步"怎么去噪"反映运动结构）
- 即使 z_gen 和 z_ref 位置差很远，只要运动方向相关就能起效
- velocity guidance 的 shift 量级天然可控
- 论文故事好讲："L2 提供起点偏置，L3 提供过程方向引导，语义正交互补"

劣势：

- 实现稍复杂（需从 trajectory 计算差分速度）
- v_ref 在高维空间中是否对 z_gen 有意义需实验验证
- 速度方向的坐标系依赖于位置，位置不同时速度方向的可迁移性不确定

#### 方案 1（备选）：Pilot Run 构建对齐轨迹

核心思想：用 blended z_T 先做一遍"无 anchor"的标准去噪，记录轨迹，再用这条轨迹做 anchor。

```python
# 第一遍（pilot）：z_T_blended → 正常去噪 → pilot_trajectory
# 第二遍（正式）：z_T_blended → anchor to pilot_trajectory → 最终视频
```

优势：起点天然对齐（cos=1.0），轨迹和生成"说同一条路"。

劣势：额外一次完整前向去噪（+1.5~2min/样本）；pilot 轨迹不携带参考视频运动信息，本质退化为 self-consistent regularization。

### B.5 实验计划

1. **方案 2 快速验证**：`--anchor_beta_max 0.03 --anchor_schedule warmup_decay`，callback 中改为 `effective_β = β_t × cos`，去掉 cos_threshold 逻辑
2. 若方案 2 ≈ 纯 L2（anchor 没起作用），转方案 3
3. 若方案 2 < 纯 L2（仍有害），说明位置 lerp 在起点不对齐时根本不可行，必须转方案 3

## 附录 C：Feature Injection (FI) 自适应门控实验（6-15）

> **实验时间**：2026-06-15 19:29~20:20
>
> **核心目标**：验证 L2(SVD) + L3(FI 特征注入) 的叠加效果。FI 在 DiT 中间层注入参考视频特征，用自适应门控动态调整注入强度。

### C.1 FI 设计原理

**特征注入（Feature Injection）**：在 Wan2.1 的 DiT transformer 中，对指定层（mid=层10~19）挂载 PyTorch forward hook，在每步去噪的前向传播中，将当前生成激活 h_gen 与预缓存的参考激活 h_ref 做加权混合：

```python
h_injected = (1 - λ_eff) * h_gen + λ_eff * h_ref
```

**自适应门控**：
```python
cos_sim = cosine_similarity(h_gen, h_ref)     # 特征余弦相似度
gate = 1.0 - sigmoid(temp * (cos_sim - 0.5))    # cos 高→gate 小
λ_eff = λ * gate * quality_scale                # 实际注入强度
```
- quality_scale：来自 η_temporal 帧间余弦相似度的软门控，运动一致性高的样本允许更强注入
- gate：特征越接近参考→注入越少，防止过度约束

### C.3 关键发现

1. **XCLIP 平均 0.8138，比纯 baseline 提升 +13.6%**——这是迄今所有方案中 XCLIP 提升最大的，远超 VDA v2 的 +3.8%
2. **样本 21 是最大赢家**：XCLIP 从 0.5195 跳到 0.8530（+64.2%），纸飞机从"几乎不可见"变为"清晰可辨"
3. **样本 31 是唯一退步的**：CLIP -6.2%，XCLIP -15.3%。η_temporal 帧间 mean_cos=0.045（低于阈值0.05），quality_scale=0.527 仍较高，但注入可能干扰了该样本的复杂水下场景
4. **quality_scale 与效果的弱相关**：样本 17 的 quality_scale 仅 0.280（微弱引导），但效果极好（XCLIP +6.4%）；样本 32 的 quality_scale 仅 0.196，效果也正（XCLIP +20.1%）。说明 FI 的自适应门控在低 quality_scale 时自动降低了注入量，避免了破坏
5. **CLIP 均值 0.9042**：比纯 baseline 0.8703 提升了 +3.9%，但低于6-11周会的SVD v1最优 0.8964+0.7874。这需要重新用相同 caption 源跑 SVD v1 10 样本做精确对比
